# 대외활동 큐레이션 - 데이터 수집

기획서에서 "정한 소스 안에서는 빠뜨리지 않는다"고 했는데, 그 "어떻게 가져올지"를 이 문서에서 정한다.
원칙은 하나다. API로 받을 수 있는 건 API로 받고, 없는 것만 크롤링한다. API가 구조화된 데이터를 주니
정확하고 안정적이고 유지보수가 싸다. 크롤링은 사이트가 바뀌면 깨지니 꼭 필요한 소스에만 쓴다.

## 소스를 둘로 나눈다

### API로 받는 것 (우선)

- 온통청년 청년정책 API. 중앙정부와 전국 지자체의 청년정책, 지원금, 장학을 한곳에 모아 구조화된 JSON으로 준다.
  data.go.kr에서 인증키를 받아 호출한다. 지자체마다 흩어진 사소한 청년 혜택까지 한 번에 들어오는 게 크다.
- 1365 자원봉사 API. 봉사 참여정보를 제공한다. 봉사 트랙은 여기서 받는다.
- API는 지역, 연령, 소득 같은 조건을 구조화된 필드로 주는 경우가 많아 규칙만으로 우리 값에 매핑된다.
  자유 문장을 LLM으로 해석하는 부담이 거의 없다.

### 크롤링하는 것 (API가 없어서)

- 링커리어, 캠퍼스픽, 콘테스트코리아. 대외활동과 공모전이 여기 많은데 공개 API가 없어 크롤링해야 한다.
- 이 중 상당수는 자바스크립트로 화면을 그리는 방식이라, 정적 HTML만으로는 안 긁혀서 브라우저를 띄우는
  방식이 필요할 수 있다. 소스마다 난이도가 다르니 하나씩 확인하며 붙인다.

## 크롤링은 어떻게 하나

말로 풀면 이렇다.

1. 목록 페이지를 한 장씩 끝까지 넘겨가며 각 공고의 기본 정보를 딴다. 제목, 카테고리, 마감일, 상세 주소, 고유 ID.
2. 각 공고 상세에 들어가 본문을 통째로 저장한다. 자격요건이 자유 문장이라 여기서 억지로 해석하지 않고
   원문 그대로 raw로 보관한다.
3. 저장한 원문을 LLM 파서에 넘겨 조건을 구조화한다. 각 조건을 원문 어디서 읽었는지 근거 문장도 같이 남긴다.
4. 이 무거운 작업은 배치로 몰아서 돌린다. 사용자는 이미 정리된 결과만 바로 조회한다.

## 중복으로 안 잡히게

두 겹으로 막는다.

- 같은 소스 안에서는 공고 고유 키로 구분한다. API는 정책·공고 ID, 크롤링은 source_url을 유일 키로 쓴다.
  이미 본 키면 기존 걸 덮어써 최신으로 맞추고, 처음 보는 키면 새로 추가한다. 배치를 몇 번 돌려도 안 쌓인다.
- 소스가 서로 다른데 같은 공고인 경우는, 제목·주최·마감을 정규화해 비교해서 하나로 묶는다.

## 안 놓치게

핵심 약속이 "커버하는 소스 안에서는 안 놓친다"라, 완전성을 이렇게 챙긴다.

- 목록을 중간에 끊지 않고 끝 페이지까지 훑는다. 새 항목이 안 나오거나 정한 최대 페이지에 닿으면 멈춘다.
- 같은 정보를 여러 소스가 다루면 교차로 받아 한쪽이 빠뜨려도 다른 쪽에서 잡는다.
- 매일 다시 받는다. 새로 올라온 공고를 그날 안에 집어넣는다.
- 개수를 감지한다. 어제보다 급감하거나 0건이면 수집이 깨진 신호로 보고 알린다.

## 마감 지난 건 자동으로 뺀다

공고는 다 기한이 있으니, 시간이 지나면 저절로 빠지게 한다.

- 마감일이 지난 지 하루 넘은 공고는 목록에서 제외한다(D+1).
- 상시·선착순처럼 마감이 없는 건 예외로 유지한다.
- 마감일을 못 읽은 건 지우지 않고 확인 필요로 둔다. 다시 받았을 때 원문에서 사라졌으면 그때 제거한다.

## 뭘 구조화하고 뭘 원문으로 두나

딱 떨어지는 값은 바로 구조화한다. 마감일, 카테고리, 지역, 연령처럼 형식이 분명한 것들. API가 구조로 주는
값도 여기 든다. 자격요건처럼 자유 문장으로 된 건 raw로 두고 LLM한테 맡긴다. 못 읽거나 애매한 건 억지로
안 채우고 확인 필요로 뺀다. 기획서에서 정한 원칙 그대로다.

## API와 크롤링에서 LLM을 어떻게 나눠 쓰나

- API가 구조로 주는 값(지역, 연령, 소득 조건 등)은 표로 우리 값에 매핑한다. LLM 없이.
  (매핑 규칙은 docs/schema.md 핵심 규칙 참고)
- LLM은 크롤링한 공고의 자유 문장 참가조건(학년, 소득, 특정 대상 등)만 맡는다.
- 제한 없는 공고는 그냥 통과로 둔다. 자격 매칭의 값어치는 조건이 걸린 공고에서 나온다.

이렇게 하면 LLM에 넘기는 양이 줄어 정확도와 비용이 같이 나아진다.

## 원문으로 연결한다

각 공고 카드에서 원문·신청 페이지로 바로 가는 링크를 둔다. 우리가 내용을 재게시하는 게 아니라
사용자를 원문으로 보내는 구조라, 소스를 많이 붙여도 법적으로 안전하다.

## 예의랑 법적 선

- 크롤링은 요청 사이에 간격을 두고, User-Agent에 우리가 누구인지 밝히고, robots.txt를 지킨다.
- robots가 허용이라고 재게시까지 허락한 건 아니다. 원문을 그대로 옮겨 담으면 약관이나 저작권 문제가 생긴다.
- 그래서 선을 이렇게 긋는다. 원문을 재게시하지 않는다. 자격 판정에 필요한 조건만 뽑아 쓰고, 실제 내용은
  항상 원문 링크로 보낸다. 출처를 표시한다. 소스를 늘릴 때마다 약관을 먼저 확인하고, 내려달라는 요청이
  오면 바로 내린다.
- API는 각 제공처의 이용 약관과 호출 한도를 지킨다.

## 정확도는 이렇게 잰다

핵심 약속이 "내 자격에 맞는 것만 골라준다"라, 파싱이 얼마나 정확한지를 숫자로 봐야 한다.

- 실제 공고 50건쯤을 뽑아, 사람이 직접 정답(이 공고는 어떤 조건인지)을 라벨링해 gold 셋을 만든다.
- 파서가 뽑은 결과를 gold랑 비교해 정밀도(가능하다고 한 게 실제로 가능한 비율)와 재현율(되는 걸 놓친 비율)을 잰다.
- 기준에 못 미치면 프롬프트랑 매핑 표를 고친다.

LLM을 실제로 돌리는 건 API 키가 있어야 해서, 키가 준비되면 측정한다. 지금은 방법까지 정해 둔다.

## 어느 소스부터 붙이나

깨끗한 API부터 붙여 데이터 흐름을 먼저 완성한다. 온통청년 청년정책 API가 첫 소스다. 구조화된 JSON이라
크롤링이나 LLM 부담 없이 파이프라인을 세울 수 있다. 그 위에 1365 봉사 API를 얹고, 크롤링 소스는
난이도 낮은 것부터 하나씩 붙인다. 수집기는 하나씩 끼우는 구조라 소스가 늘어도 나머지는 안 건드린다.

## 크롤링 소스 붙일 때 확정할 것 (세부)

- 중복 키는 source_url을 유일 키로 쓴다. 공고 고유 번호가 URL 안에 있으면 뽑아 참고용으로 같이 저장한다.
  (schema.md의 source_url unique와 맞춘다.)
- 페이지네이션은 목록 주소의 페이지 값을 1부터 늘린다. 새 항목이 안 나오거나 정한 최대 페이지에 닿으면 멈춘다.
- 상세에서 뽑아 어디에 넣을지.
  - 제목 -> raw_title
  - 카테고리 태그 -> schema.md 매핑표로 majors
  - 대상(대학생, 일반인 등) -> 참고용. 대학생 여부로 재학상태 힌트만
  - 접수기간 끝 날짜 -> deadline
  - 참가조건 자유 문장 전체 -> raw_text (LLM이 학년, 소득, 특정 대상 등을 뽑음)
  - 상세 주소 -> source_url
- 마감일 파싱은 접수기간이 "YYYY-MM-DD ~ YYYY-MM-DD" 꼴이면 끝 날짜를 deadline으로 쓴다. D-day만 있으면
  수집 시각 기준으로 계산한다. 둘 다 못 읽으면 deadline은 null로 두고 확인 필요로 보낸다.

## 개발 환경은 (방향)

스택은 프론트 React, 백 Express, 데이터는 Supabase 방향이다. 남이 짜준 걸 그대로 붙이는 게 아니라
자바스크립트랑 CSS 기본을 같이 짚으면서, 내가 이해하고 넘어간다.
