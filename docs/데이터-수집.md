# 대외활동 큐레이션 - 데이터 수집

"어떤 소스에서 어떻게 가져올지"를 정하는 문서다. 원칙은 둘이다.
하나, API로 받을 수 있으면 API로 받고 없으면 크롤링한다. API가 구조화된 데이터를 주니 정확하고 안정적이다.
둘, **최대한 많은 소스에서 빠짐없이 총망라한다. 이게 이 서비스의 핵심이자 1순위다.** 그렇게 다 모은 것 중에서
내 자격에 맞는 걸 정확히 골라 위로 올린다. 안 맞아도 버리지 않고, 못 읽은 조건은 확인 필요로 정직하게 둔다.
즉 다 모으는 것(총망라)과 잘 골라주는 것(정확한 매칭)은 대립이 아니라, 수집 단계와 표시 단계로 층이 다르다.

## 소스 (확정)

대외활동·공모전이 핵심이고, 장학·봉사·지자체 혜택이 보조다.

- **링커리어** — 대외활동·서포터즈·공모전의 핵심. 여기는 못 놓친다. SPA라 화면 HTML은 어렵지만,
  SPA는 화면을 그리려고 내부 JSON API를 부르니 그 API를 직접 받는다(아래 참고). 없으면 그때 헤드리스로.
- **콘테스트코리아** — 공모전이 많다. 크롤링으로 받는다.
- **온통청년 청년정책 API** — 중앙·지자체 청년정책·지원금. 특히 지자체 청년 혜택(부산 등)이 중요하다.
  공공데이터포털 인증키로 받는다. 실제 연동 완료(지역이 구조 필드로 옴, zipCd로 부산/금정구 판별).
  화면에서는 카테고리 "지자체"로 분류한다.
- **한국장학재단 학자금지원정보 API** — 장학금. (공공데이터포털 15028252)
- **1365 자원봉사 API** — 봉사. (공공데이터포털 15000221)

## 링커리어는 내부 API로 받는다

링커리어는 자바스크립트로 화면을 그리는 SPA다. 화면 HTML을 긁으려 하면 어렵지만, SPA는 데이터를
자기 서버의 JSON API에서 받아온다. 그 요청을 직접 부르면 헤드리스 브라우저 없이 깔끔한 JSON을 받는다.

- 브라우저 개발자도구 Network > Fetch/XHR 에서 목록을 넘길 때 뜨는 요청을 확인한다.
- JSON으로 공고 목록을 주면 그게 우리가 쓸 소스다. 구조화돼 있어 HTML 파싱보다 안정적이다.
- 다만 비공식 API라 링커리어가 막거나 바꿀 수 있다. 요청 간격을 지키고, 약관·robots를 확인한다.

## 콘테스트코리아 크롤링

1. 목록 페이지를 끝 페이지까지 넘기며 제목·카테고리·마감일·상세 주소·고유 ID를 딴다.
2. 상세 본문을 통째로 raw로 저장한다. 자격요건이 자유 문장이라 여기서 해석하지 않는다.
3. 저장한 원문을 LLM에 넘겨 조건을 구조화한다(근거 문장도 같이 남긴다).

## 비용은 극단적으로 줄인다

LLM 비용이 이 프로젝트에서 유일하게 돈 드는 부분이라 최소로 잡는다. 하루 1달러도 비싸다.

- **매일 배치를 돌리지 않는다.** 혼자 쓰는 도구를 24시간 굴릴 이유가 없다.
- **내가 열 때 신규 공고만 파싱한다(온디맨드).** 이미 파싱한 공고는 다시 안 돌린다.
- **저렴한 모델(Claude Haiku)** 을 쓰고, 프롬프트에서 반복되는 지시문은 캐싱한다.
- 구조로 오는 값(API의 지역·소득)은 규칙으로 매핑해 LLM을 아예 안 거친다. LLM은 크롤링한 자유 문장만.
- 초기에 한 번 전량 파싱하고, 그 뒤로는 하루 신규 몇 건이라 비용이 거의 안 나온다. 정확한 값은 실측해서 관리한다.

## 언제 받아오나

실시간으로 소스를 계속 때리지 않는다(차단·부하). 하루 1~2회, 또는 내가 열었을 때 마지막 수집이 오래됐으면
그때 신규분만 받아온다. 링커리어 내부 API도 마찬가지로 아껴서 호출한다.

## 중복으로 안 잡히게

두 겹으로 막는다.

- 같은 소스 안에서는 고유 키로 구분한다. API는 공고 ID, 크롤링은 source_url을 유일 키로 upsert 한다.
- 소스가 다른데 같은 공고면 제목·주최·마감을 정규화해 하나로 묶는다.

## 안 놓치게 (이 서비스의 핵심)

빠짐없이 모으는 게 제일 중요하다. 소스를 최대한 늘리고, 각 소스 안에서 하나도 안 놓친다. 커버하는 소스는
공개해서 어디까지 훑는지 정직하게 밝힌다.

- 목록을 중간에 끊지 않고 끝까지 훑는다.
- 소스를 계속 늘린다. API로 받을 수 있는 건 다 받고, 없으면 크롤링으로 붙인다.
- 여러 소스가 같은 공고를 다루면 교차로 받아 한쪽이 빠뜨려도 다른 쪽에서 잡는다.
- 개수를 감지해 0건이거나 급감하면 수집이 깨진 신호로 본다.

## 마감 지난 건 뺀다

- 조회할 때 마감 지난 공고는 목록에서 거른다(이게 기본). 별도 배치 없이 조회 필터만으로 충분하다.
- 상시·선착순처럼 마감 없는 건 유지한다.
- 마감일을 못 읽은 건 지우지 않고 확인 필요로 둔다.

## 뭘 구조화하고 뭘 확인 필요로 두나

딱 떨어지는 값(마감·카테고리·지역)은 구조화한다. 자유 문장 자격요건은 LLM이 뽑되, 못 읽거나 애매한
조건만 "확인 필요"로 남긴다. **공고 전체를 확인 필요로 밀지 않고, 읽어낸 조건은 판정하고 못 읽은 것만 표시한다**
(부분 판정). 그래야 확인 필요가 화면을 도배하지 않는다.

## 원문으로 연결한다

각 공고 카드·상세에서 원문·신청 페이지로 바로 가는 링크를 둔다. 내용을 재게시하지 않고 사용자를 원문으로
보내는 구조라, 소스를 붙여도 법적으로 안전하다.

## 예의랑 법적 선

- 크롤링·내부 API 호출은 요청 사이 간격을 두고, User-Agent를 밝히고, robots.txt와 약관을 지킨다.
- 원문을 재게시하지 않는다. 자격 판정에 필요한 조건만 뽑아 쓰고, 실제 내용은 항상 원문 링크로 보낸다.
- 출처를 표시하고, 내려달라는 요청이 오면 바로 내린다.
- API는 각 제공처의 이용 약관과 호출 한도를 지킨다.

## 성공을 이렇게 잰다

총망라가 목표라 "얼마나 빠짐없이 모으느냐"가 1순위다.

- 주력 지표: 커버하는 소스 수와 소스별 수집 개수. 소스를 늘리고 각 소스에서 다 긁는 걸로 키운다.
- 놓침률(원천 전체 대비 빠뜨린 비율)은 전체를 몰라 정확히는 못 재니, 소스 한 곳을 손으로 세어 샘플 대조로 근사한다.
- 차별점 지표(병행): 얼마나 정확히 골라주느냐. 실제 공고 30~50건을 사람이 직접 라벨링해 gold 셋을 만들고,
  파서·매칭 결과와 비교해 정밀도·재현율을 잰다. 기준 미달이면 프롬프트와 매핑 표를 고친다.

## 먼저 찔러볼 것 (제일 불확실한 것부터)

쉬운 것부터가 아니라 제일 불확실한 것부터 확인한다.

- 링커리어 내부 API가 실제로 있는지 Network 탭에서 확인한다. 있으면 JSON 몇 건 받아본다.
- 콘코 상세 한 건을 Claude에 넣어 자격조건이 제대로 뽑히는지 본다. **여기가 이 프로젝트의 심장이다.**
  여기서 되면 나머지는 다 된다. 안 되면 기획을 일찍 고칠 수 있다.

## 크롤링/수집 세부

- 중복 키: 크롤링은 source_url, API는 공고 ID.
- 페이지네이션: 목록 주소의 페이지 값을 1부터 늘리고, 새 항목이 없거나 최대 페이지에 닿으면 멈춘다.
- 상세에서 뽑아 넣을 곳: 제목 -> raw_title, 카테고리 -> majors 매핑 후보, 접수기간 끝 -> deadline,
  참가조건 자유 문장 -> raw_text, 상세 주소 -> source_url.
- 마감일: "YYYY-MM-DD ~ YYYY-MM-DD"면 끝 날짜, D-day만 있으면 수집 시각 기준 계산, 둘 다 못 읽으면 null + 확인 필요.

## 개발 스택

프론트 React, 데이터 Supabase(프론트가 직접 조회, Express 조회 서버 없음). 남이 짜준 걸 붙이지 않고 기본을 짚으며 이해하고 넘어간다.
