# 대외활동 큐레이션 - 데이터 수집

기획서에서 "정한 소스 안에서는 빠뜨리지 않는다"고 했는데, 그 "어떻게 긁어올지"를 이 문서에서 확실히 정한다. 추측으로 적은 게 아니라 실제로 위비티를 열어보고 확인한 내용이다.

## 확인한 것 (직접 열어봄)

- 위비티 공모전 목록은 정적 HTML이다. 제목, 카테고리, 마감일(D-day), 상세 페이지 링크가 자바스크립트 없이 HTML에 그대로 들어 있다. 브라우저를 띄워서 화면을 그릴 필요가 없다는 뜻이라, 가볍고 빠르게 긁을 수 있다.
- 상세 페이지도 마찬가지다. 응모대상, 주최, 시상, 접수기간 같은 원문이 정적 HTML 안에 있다.
- 공고마다 고유 번호(ix)가 붙는다. 예를 들어 ix=108509. 이 번호를 그 공고의 신분증처럼 쓴다.
- robots.txt가 전체 허용이다. 다만 예의상 요청 사이에 3초 정도 쉰다.

## 어떻게 긁나

말로 풀면 이렇다.

1. 목록 페이지를 한 장씩 넘겨가며 각 공고의 기본 정보를 딴다. 제목, 카테고리, 마감일, 상세 페이지 주소, ix 번호.
2. 각 공고의 상세 페이지에 들어가 본문을 통째로 저장한다. 응모대상이나 자격요건이 자유 문장으로 적혀 있어서, 여기서 억지로 해석하지 않고 원문 그대로 raw로 보관한다.
3. 저장한 원문을 LLM 파서에 넘겨 조건을 구조화한다. 이때 각 조건을 원문 어디서 읽었는지 근거 문장도 같이 남긴다. 기획서의 AI 파싱 부분이랑 이어지는 단계다.
4. 이 무거운 작업은 새벽에 한 번만 배치로 돌린다. 사용자는 이미 정리된 결과만 바로 조회한다.

## 같은 공고를 또 긁으면

ix 번호로 구분한다. 이미 본 ix면 기존 걸 덮어써서 최신으로 맞추고, 처음 보는 ix면 새 공고로 추가한다. 그래서 배치를 몇 번 돌려도 중복이 안 쌓인다.

## 뭘 구조화하고 뭘 원문으로 두나

딱 떨어지는 값은 바로 구조화한다. 마감일이나 카테고리처럼 형식이 분명한 것들. 자격요건처럼 자유 문장으로 된 건 raw로 두고 LLM한테 맡긴다. 못 읽거나 애매한 건 억지로 안 채우고 확인 필요로 뺀다. 이건 기획서에서 정한 원칙 그대로다.

## 깨지면 어떻게 아나

사이트 구조가 바뀌면 수집이 조용히 망가질 수 있다. 그래서 목록이 갑자기 0건이 되거나 특정 필드가 대부분 비어서 나오면 이상 신호로 보고 알림을 띄운다. 소스는 하나씩 분리돼 있어서, 위비티가 깨지면 위비티 수집기만 고치면 된다.

## 예의랑 법적 선

- 요청 사이에 3초 정도 쉰다. 상대 서버에 부담 안 주려고.
- 우리가 누구인지 User-Agent에 밝힌다.
- robots.txt를 지킨다. 다만 robots가 허용이라고 재게시까지 허락한 건 아니다. 위비티도 자기가 공고를 모아 보여주는 서비스라, 그 데이터를 그대로 우리 서비스로 옮겨 담으면 약관이나 저작권 문제가 생길 수 있다.
- 그래서 선을 이렇게 긋는다. 원문을 재게시하지 않는다. 자격 판정에 필요한 조건만 뽑아 쓰고, 실제 공고 내용은 항상 원문 링크로 보낸다. 출처(위비티)를 표시한다. 소스를 늘릴 때마다 그 사이트 약관을 먼저 확인하고, 내려달라는 요청이 오면 바로 내린다.

## 실데이터에서 더 확인한 것 (위비티 구조 활용)

위비티 상세를 여러 건 열어보니 자유 문장만 있는 게 아니었다. 위비티가 분야(광고/마케팅, 웹/모바일/IT, 과학/공학, 예체능/미술/음악 등)와 응모대상(대학생, 일반인, 청소년 등)을 자체 분류로 이미 붙여 둔다. 그리고 첫 페이지에는 자격 제한이 없는 "제한없음, 누구나 참여" 공고가 흔했다.

그래서 수집과 파싱을 이렇게 나눈다.

- 위비티가 구조로 주는 분야나 응모대상 같은 건 표로 우리 값에 매핑한다. LLM 없이. (매핑 규칙은 docs/schema.md 핵심 규칙 참고)
- LLM은 나머지, 참가조건에 자유 문장으로 적힌 세부(학년, 소득, 특정 대상 등)만 맡는다.
- 제한 없는 공고는 그냥 통과로 둔다. 자격 매칭의 값어치는 조건이 걸린 공고에서 나온다.

이렇게 하면 LLM에 넘기는 양이 줄어서 정확도와 비용이 같이 나아진다.

## 정확도는 이렇게 잰다

핵심 약속이 "내 자격에 맞는 것만 골라준다"라서, 파싱이 얼마나 정확한지를 숫자로 봐야 한다. 안 그러면 근거 없는 말이다.

- 실제 위비티 공고 50건쯤을 뽑아, 사람이 직접 정답(이 공고는 어떤 조건인지)을 라벨링해 gold 셋을 만든다.
- 파서가 뽑은 결과를 gold랑 비교해 정밀도(가능하다고 한 게 실제로 가능한 비율)와 재현율(되는 걸 놓친 비율)을 잰다.
- 기준에 못 미치면 프롬프트랑 매핑 표를 고친다.

LLM을 실제로 돌리는 건 API 키가 있어야 해서, 키가 준비되면 측정한다. 오늘은 방법까지 정해 둔다.

## 소스 늘리기

일단 위비티 하나로 끝까지 돌려본다. 위비티는 정적 HTML이라 쉬운 편이다. 링커리어처럼 자바스크립트로 화면을 그리는 사이트는 정적 방식으로는 안 긁혀서, 나중에 브라우저를 띄우는 방식으로 따로 붙인다. 수집기는 하나씩 끼우는 구조라 소스가 늘어도 나머지는 안 건드린다.

## 다음주 시작 전에 확정 (스크래핑 세부)

- 중복 키는 source_url을 유일 키로 쓴다. source_url 안에 ix가 들어 있으니 ix는 거기서 뽑아 참고용으로 같이 저장한다. (schema.md의 source_url unique와 맞춘다. 문서끼리 어긋났던 것 정리함.)
- 페이지네이션은 목록 주소의 gp 값을 1부터 늘린다(gp=1, 2, 3 ...). 새 ix가 안 나오는 페이지가 나오거나 정한 최대 페이지에 닿으면 멈춘다.
- 위비티 상세에서 뽑아 어디에 넣을지.
  - 제목 -> raw_title
  - 분야(카테고리 태그) -> schema.md 매핑표로 majors
  - 응모대상(대학생, 일반인 등) -> 참고용. 대학생 여부로 재학상태 힌트만
  - 접수기간 끝 날짜 -> deadline
  - 참가조건 자유 문장 전체 -> raw_text (LLM이 학년, 소득, 특정 대상 등을 뽑음)
  - 상세 주소 -> source_url
- 마감일 파싱은 접수기간이 "YYYY-MM-DD ~ YYYY-MM-DD" 꼴이면 끝 날짜를 deadline으로 쓴다. D-day만 있으면 수집 시각 기준으로 계산한다. 둘 다 못 읽으면 deadline은 null로 두고 확인 필요로 보낸다.

## 개발 환경은 (오늘은 방향만)

개발은 2주차에 한다. 스택은 프론트 React, 백 Express, 데이터는 Supabase 방향으로 보고 있다. 오늘은 코드까지 안 짠다. 나중에 만들 때는 자바스크립트랑 CSS 기본을 같이 짚으면서 간다. 남이 짜준 걸 그대로 붙이는 게 아니라 내가 이해하고 넘어가려고.
