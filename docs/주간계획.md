# 2주차 주간 계획

이번 주에 하려는 건 화면부터 DB까지 한 바퀴가 실제로 도는 걸 만드는 거다. React 화면에서 조건을 넣으면,
Express 서버가 Supabase를 조회해서 매칭한 결과를 다시 화면으로 돌려주는 한 사이클. 이걸 수직 슬라이스라고 한다.
완성된 기능이 아니라 프로토타입으로, 이 흐름이 진짜 도는지를 이번 주에 눈으로 확인하는 게 목표다.

스택은 정한 대로 간다. 프론트는 React(Vite), 백은 Express, 데이터는 Supabase. 이번 주 화면은 React로 짠다.
발표 데모는 HTML 한 장이었지만, 실제 개발은 React 환경에서 한다.

데이터는 처음엔 시드 몇 건으로 시작한다. 화면과 서버와 DB가 연결되는지 보는 데는 진짜 데이터가 많을 필요가 없다.
연결이 되고 나면 온통청년 청년정책 API를 한 소스 붙여서 실데이터도 흘려본다. 이번 주 핵심은 데이터 양이 아니라
FE와 BE와 DB가 이어지는 것이다.

전체 백로그는 [checklist.md](checklist.md), 설계는 [schema.md](schema.md), 수집 전략은 [데이터-수집.md](데이터-수집.md)에 있고,
이번 주 작업은 GitHub 이슈 #1부터 #7로 쪼개 뒀다.

## 요일별로

### 월요일 - DB부터 세운다 (이슈 #1)
Supabase 프로젝트를 만들고 schema.md대로 테이블을 만든다. 화면과 서버를 테스트할 수 있게 postings에 시드를
몇 건 손으로 넣는다. server에서 그 시드가 조회되면 그날 몫은 끝이다. 올리기 전에 security-check 스킬로 .env가
안 새는지 본다.

### 화요일 - 서버 통로를 연다 (이슈 #2)
Express 서버를 띄우고 GET /postings를 만든다. Supabase에서 목록을 읽어 JSON으로 돌려주는 것까지. React에서
부를 수 있게 CORS를 열어 둔다. curl이나 브라우저로 시드가 JSON으로 나오면 됐다.

### 수요일 - React 화면 앞단 (이슈 #5)
Vite로 React 앱을 세우고 조건 입력 화면을 만든다. 학년, 전공 여덟 개, 지역 열세 개, 나이, 재학 상태, 그리고
장학을 볼 때만 소득과 학점. 각 항목에 선택 안 함을 둔다. 고른 값이 상태로 잡혀서 화면에 보이면 된다.

### 목요일 - 뒷단을 붙여 한 바퀴 돌린다 (이슈 #6, #3)
결과 화면에서 GET /postings를 불러 카드로 보여준다. 로딩과 빈 상태와 에러를 다 처리하고, 카드마다 원문으로
가는 링크를 둔다. 서버 쪽에는 매칭을 붙여서 가능, 확인 필요, 거의 가능으로 나눠 준다. 여기까지 되면 화면에서
넣은 조건이 서버를 거쳐 DB를 조회하고 다시 화면으로 돌아오는 한 사이클이 완성된다. 이번 주의 고비다.

### 금요일 - 점검하고, 여유되면 실데이터 (이슈 #7, #4)
feature-verifier 스킬로 한 사이클을 end-to-end로 점검한다. 화면에 조건을 넣어 서버와 DB를 거쳐 결과가 제대로
분류돼 뜨는지, DB에는 진짜 들어갔는지 직접 조회해 확인한다. 시간이 남으면 온통청년 API를 한 소스 붙여 시드 대신
실데이터를 흘려본다. API 키는 발급에 며칠 걸릴 수 있어 월요일에 미리 신청해 둔다.

## 점검

매일 저녁 그날 이슈의 완료 기준을 만족했는지 본다. 못 끝냈으면 다음 날로 넘기고 뒷 일정을 당긴다. 화요일은
가볍고 목요일이 무거우니 유연하게 조정한다. 금요일에 화면에서 조건을 넣어 결과가 뜨는 한 사이클이 돌면
이번 주는 성공이다.

## 이번 주에 안 하는 것

크롤링 소스(링커리어, 캠퍼스픽, 콘테스트코리아)와 거기서 나오는 어려운 것들은 다음 주부터다. 소스 간 중복 제거,
자유 문장 LLM 파싱, 놓치지 않게 하는 장치는 그때 붙인다. 이번 주는 깨끗한 데이터로 FE와 BE와 DB를 잇는 것부터
확실히 하고, 그 위에 데이터 소스를 넓힌다. 매일 자동으로 도는 배치도 배포 단계에서 붙이고, 이번 주는 손으로
한 번 끝까지 돌리는 데까지만 한다.

## 규칙

커밋은 작게 쪼갠다. 한 커밋은 한 가지 일만 하고, 메시지에 무엇을 왜 했는지 적는다. 올리기 전에 security-check
스킬로 비밀키와 개인정보와 .env를 검사한다. 마감이 지난 지 하루 넘은 공고는 뺀다(상시와 선착순은 예외). 공고
내용을 재게시하지 않고 상세와 신청은 항상 원문 링크로 보낸다.
