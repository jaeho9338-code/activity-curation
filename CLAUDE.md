# CLAUDE.md - 대외활동 큐레이션

내 조건에 맞는 대외활동, 공모전, 장학, 봉사만 골라주는 서비스. 이 파일은 개발할 때 참고할 맥락이다.
개발은 2주차에 본격적으로 하고, 지금은 환경과 규칙만 정해 둔다.

관련 문서
- 기획서(위키): https://github.com/jaeho9338-code/activity-curation/wiki
- 데이터 수집 방식: docs/데이터-수집.md
- 자격 스키마 / API 계약: docs/schema.md
- 디자인 시스템: design/시안.html 과 .claude/skills/curation-design 스킬

## 스택

- 프론트: React (Vite). Supabase를 직접 조회(supabase-js), Express 조회 서버는 안 만든다.
- 데이터: Supabase (Postgres, 자격요건은 JSONB로)
- 수집: API 우선(온통청년 청년정책 등), 없으면 정적 HTML 스크래핑/내부 API(링커리어). Node 스크립트로 온디맨드 수집.
- 파싱: Claude (Anthropic, 자격요건 자유 문장을 조건으로 구조화)

## 디렉토리 구조

```
activity-curation/
  src/             React 앱 (Vite). 지금은 root가 곧 client다.
  server/          Node 스크립트 (Express 서버 아님, 조회는 프론트가 Supabase 직접)
    sources/       소스별 수집기 (콘테스트코리아, 위비티, 링커리어, 부산청년플랫폼, 온통청년)
    parser/        LLM 자격요건 파서
    batch/         자동 수집기 (collect.js, 온디맨드/GitHub Actions)
  docs/            기획서, 데이터 수집, 스키마
  design/          디자인 시안
  .claude/skills/  디자인 스킬
```

프론트를 client/ 폴더로 옮기는 건 나중에. 프로토타입 단계에선 root를 client로 둔다.

## 라이브러리 (왜 쓰는지)

- 프론트: react, react-dom, react-router-dom(화면 이동). 스타일은 디자인 토큰 CSS로 가볍게, UI 라이브러리는 안 쓴다.
- 서버 스크립트: dotenv(키 관리), @supabase/supabase-js(DB), cheerio(정적 HTML 파싱), @anthropic-ai/sdk(Claude 파싱).
- 개발: vite, oxlint(이미 씀).
- 원칙: 꼭 필요한 것만 넣는다. 안 쓸 라이브러리를 미리 안 깐다.

## 컨벤션

- 컴포넌트 파일은 PascalCase, 나머지는 소문자.
- 색과 간격은 디자인 토큰만 쓴다. 임의 색 금지. (.claude/skills/curation-design 의 색 토큰 참조)
- API는 docs/schema.md 의 계약을 따른다.
- 화면은 로딩, 빈 상태, 에러 상태를 항상 처리한다.
- 비밀키는 .env 로만 관리한다. 절대 커밋하지 않는다.

## 커밋 규칙

- 작게 쪼갠다. 한 커밋은 한 가지 일만.
- 메시지는 한글로, 무엇을 왜 했는지 적는다. 긴 대시 기호는 안 쓴다.
- 개인정보(이름, 전화, 이메일)와 키는 커밋하지 않는다.

## 개발할 때

- 남이 짜준 걸 그대로 붙이지 않는다. 자바스크립트와 CSS 기본을 짚으며 이해하고 넘어간다.
- 서두르지 않는다. 핵심인 자격 매칭부터 되는지 확인하고 넓힌다.

## 확정한 설계 결정 (다음주 생각 없이 시작하려고)

- 조회는 프론트가 Supabase에서 직접 한다(supabase-js). Express 조회 서버는 안 만든다(개인 규모라 불필요). 매칭도 화면(클라)에서 match.js로 한다. 공고 수천 건은 브라우저 메모리에서 필터해도 감당된다. 수집·파싱만 Node 스크립트(온디맨드)로 돌려 Supabase에 write. 데이터 흐름은 docs/schema.md "데이터 흐름" 참고.
- 신선도: 매일 배치를 돌리지 않는다. 내가 열 때 마지막 수집이 오래됐으면 그때 신규분만 받거나(온디맨드), 하루 1~2회만 받는다. 카드에 수집 시각을 표시하고 마감 임박을 강조한다.
- LLM은 Claude를 쓴다. 비용은 신규·변경분만 파싱하고, API가 구조로 주는 값(지역, 소득 등)은 규칙으로 매핑해 호출을 줄인다. 크롤링한 자유 문장만 저렴한 모델(Haiku급)로 처리하되, 원래 분류·추출을 2콜로 나눴던 걸 1콜(tool use 하나에 합침)로 바꿨다 — 호출 수와 반복되는 프롬프트 지시문이 절반으로 줄어 비용도 절반이 됐다(실측 건당 $0.0022). 한 번 파싱한 공고는 다시 안 돌린다. 정확한 비용은 실측한다.
- 로그인은 없다. 프로필(내 조건)은 브라우저 상태와 로컬스토리지로만 두고 DB에 저장하지 않는다. 혼자·지인 위주로 쓰는 규모라 계정·인증을 안 만든다. 즐겨찾기도 로컬스토리지로 저장한다.
- 매칭에서 "거의 가능"(조건 1개 차이)은 학년만 1급간 차이까지 근접으로 보고, 나머지 조건은 맞다/틀리다로 본다(학점 0.1 같은 미세 차이는 안 따진다).
- 카테고리(공모전·대외활동·장학·봉사)는 탭으로 딱 나누지 않고 태그로 달아 필터한다. 봉사·장학이 대외활동 명분과 겹치는 등 경계가 모호해서다.
- 총망라(빠짐없는 수집)가 최우선 니즈다. 소스를 최대한 많이 모으고 각 소스에서 안 놓친다. 다 모은 것 중 자격 맞는 걸 정확히 골라 위로 올린다(정확한 매칭이 차별점). 안 맞아도 안 버리고 확인 필요/불가로 둔다. 못 읽은 조건은 공고 전체가 아니라 그 조건만 확인 필요로 표시한다(부분 판정). 총망라와 정확도는 대립이 아니라 수집·표시 층이 다른 것.
- 소스: 링커리어(대외활동·공모전 핵심, 내부 API), 콘테스트코리아·위비티(공모전, 크롤링), 온통청년(지자체 청년정책·지원금), 부산청년플랫폼·서울 청년몽땅정보통(지자체 모집공지, 크롤링), 한국장학재단(장학, API). 1365 봉사는 안 붙인다(맞는 API 없음). 세부는 docs/데이터-수집.md.
- 스키마는 단순하게. 개인 4주 프로젝트라 테이블을 최소로 둔다(수집 원문 + 완성본 정도). 대규모 서비스식 다중 테이블·스키마 버전관리는 피한다.
- 성공 지표는 총망라가 목표라 커버 소스 수·소스별 수집 개수가 1순위. 매칭 정확도(gold 30~50건 정밀도·재현율)는 차별점 지표로 병행. 놓침률은 원천 전체를 몰라 샘플 대조로 근사한다.
- LLM 비용은 극단적으로 줄인다. 매일 배치 없이 온디맨드로 신규만, Haiku, 프롬프트 캐싱. 하루 1달러도 비싸다고 보고 실측으로 관리한다.
