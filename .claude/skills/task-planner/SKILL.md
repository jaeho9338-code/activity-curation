# 계획 수립

목표를 주면 그걸 이룰 작업으로 나누고 순서를 매긴다. 즉흥으로 개발하면 방향을 잃으니, 미리
"무엇을, 어떤 순서로, 언제 끝낼지"를 정해두는 게 목적이다.

## 나누는 순서

1. **목표부터 확인한다.** 이번 주(또는 이 기능)에서 무엇이 되어 있어야 끝인지 한 줄로 적는다.
2. **하루 크기로 쪼갠다.** 큰 덩어리는 하루 안에 끝낼 만한 크기로 나눈다. 하루보다 크면 더 쪼갠다.
3. **핵심 시나리오부터 앞에 둔다.** 없으면 서비스가 성립 안 하는 것(P0)을 먼저. 신뢰·완성도(P1),
   나중에 해도 되는 것(P2) 순으로.
4. **의존 순서를 맞춘다.** 앞 작업이 끝나야 다음이 되는 것끼리 순서를 맞춘다. 데이터 -> 로직 -> 화면 식.
5. **각 작업에 완료 기준을 붙인다.** "무엇이 되면 끝인지"를 측정 가능하게 적는다. ("됐다" 말고 "테이블 5개가 생성됨")
6. **요일이나 순번에 배치한다.**

## 출력 형식

표로 준다. 열: 작업 / 완료 기준 / 우선순위 / (필요하면) 요일이나 순번.

## 점검 체크리스트 (이미 세운 계획을 볼 때)

- 각 작업이 하루 크기인가. 너무 크면 쪼갠다.
- 완료 기준이 측정 가능한가. "됐다"처럼 애매하면 다시 쓴다.
- 핵심(P0)이 앞에 있는가.
- 앞뒤 의존 순서가 맞는가. 앞이 안 되면 뒤가 막히지 않나.
- 이번 주 목표를 이 작업들을 다 하면 실제로 이루나. 빠진 게 없나.
