# 기능 검증

만든 게 진짜 되는지 눈으로 확인하는 도구다. 코드만 읽고 "될 것이다"라고 넘기지 않는다. 실제로 돌려보고,
요구대로 나오는지 하나씩 대조한다.

## 점검 순서

1. **완료 기준부터 확정한다.** 이 기능이 무엇이 되면 통과인지 한 줄로 적는다. 없으면 먼저 만든다.
   (기준은 docs/주간계획.md의 요일별 완료 기준이나 GitHub 이슈의 완료 기준을 본다.)
2. **실제로 돌린다.** 서버를 켜고, 화면을 띄우고, 요청을 실제로 보낸다. 코드만 보고 판단하지 않는다.
3. **정상 경로.** 대표 입력 하나로 기대한 출력이 나오는지 본다.
4. **경계와 예외.** 빈 입력, 없는 데이터, 잘못된 값, 로딩과 에러 상태가 처리되는지 본다.
5. **한 사이클 확인 (수직 슬라이스).** 화면에서 요청 -> 서버가 처리 -> DB에 저장·조회 -> 응답 -> 화면이 바뀌는
   한 바퀴가 실제로 도는지 본다. DB에 진짜 들어갔는지는 DB를 직접 조회해서 확인한다.
6. **완료 기준 대조.** 위에서 정한 기준을 체크리스트로 하나씩 O/X 매긴다.
7. **결과 보고.** 통과한 것과 실패한 것을 적는다. 실패는 무엇이, 왜, 어떻게 재현되는지까지. 통과 못 했으면
   고칠 것을 짚는다.

## 원칙

- "됐다"는 말 금지. 눈으로 확인한 것만 통과로 친다.
- 실패를 숨기지 않는다. 안 되는 건 안 된다고 그대로 보고한다.
- 작은 단위로 자주 검증한다. 다 만들고 한 번에 보지 말고, 단계마다 본다.

## 범용 점검 체크리스트

- 정상 경로가 통과하나.
- 빈 값과 에러 상태가 처리되나.
- DB에 실제로 반영됐나 (직접 조회해 확인).
- 완료 기준을 다 충족하나.
- 콘솔이나 서버 로그에 에러가 없나.
