# 테스트 먼저 쓰기 (TDD)

구현부터 하고 나중에 "잘 되겠지" 하고 넘기지 않는다. 테스트를 먼저 써서 실패를 눈으로 보고(빨강),
그다음 통과하도록 구현한다(초록). 이 순서라야 테스트가 진짜로 뭔가를 검증하는지 확신할 수 있다.

이 프로젝트는 `node:test`(Node 22 내장, 새 의존성 없음)를 쓴다. 파일 이름은 `*.test.js`, 실행은
`cd server && node --test` 또는 `npm test`. 실제 예시는 `server/scripts/score.test.js`,
`server/regionLookup.test.js`, `server/sources/youthcenter.test.js`.

## 순서

1. **무엇을 검증할지 한 줄로 정한다.** "이 함수는 무슨 입력에 무슨 출력을 내야 하는가"를 먼저 적는다.
2. **테스트를 먼저 쓴다.** 함수가 아직 없어도 import부터 하고, 대표 케이스 + 경계 케이스(빈 값, null,
   극단값, 겹치는 경우)를 assert로 적는다. 값이 "안 깨지나"가 아니라 "정확히 맞나"를 본다.
3. **실패를 확인한다(빨강).** `node --test`를 돌려서 실제로 실패하는 걸 눈으로 본다. 여기서 통과하면
   테스트가 아무것도 안 하는 것이니, 테스트가 잘못된 것이다.
4. **구현한다(초록).** 테스트를 통과하도록 함수를 짠다. 테스트에 맞추려고 억지로 특수 케이스를 박지 말고,
   정직하게 로직으로 통과시킨다.
5. **다시 돌려서 통과를 확인한다.** 전부 초록이면 끝. 커밋은 빨강(테스트)과 초록(구현)을 나눠서 남기면,
   나중에 TDD 흐름이 히스토리에 그대로 보인다.

## 무엇을 테스트하나

- **순수 함수 우선.** 입력을 주면 출력이 정해지는 함수(지역 매핑, 나이 파싱, 채점 계산 등)가 테스트하기
   제일 쉽고 이득이 크다. DB·네트워크·화면이 얽힌 건 뒤로 미룬다.
- **규칙을 고칠 때 회귀 테스트.** 버그를 찾아 규칙을 고쳤으면, 그 케이스를 테스트로 박아둔다. 다음에
   코드를 바꿔도 같은 버그가 다시 안 생긴다.
- **이미 완성돼서 잘 도는 함수에 지금 테스트를 쓰는 건 TDD가 아니라 회귀 테스트다.** 그것도 가치 있지만,
   "빨강" 단계가 없으니 TDD 예시로 내세우지 않는다.

## 점검 체크리스트

- 테스트를 구현보다 먼저 썼나. 실패(빨강)를 실제로 봤나.
- 정상 케이스만이 아니라 경계·극단·겹치는 경우도 넣었나.
- "안 깨짐"이 아니라 "값이 정확함"을 assert 했나.
- 테스트가 통과하도록 구현을 정직하게 짰나(특수 케이스 박기로 우회 안 했나).
- 버그를 고친 거라면, 그 케이스가 회귀 테스트로 남았나.
